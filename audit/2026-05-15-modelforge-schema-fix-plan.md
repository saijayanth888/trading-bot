## Root Cause

There are **two independent problems**:

**Problem 1 â€” Stale SQL from the running mf-api container.** The five column-not-found errors (`has_adapter`, `id` on generations, `status` on generations, `cron_schedule`, `benchmark_name`, `ts`) are all produced by `mf-api` itself (confirmed: all idle DB connections belong to `172.20.0.5 = mf-api`). The current host source tree (`apps/api/src/`) does NOT contain these queries â€” they exist only in the **compiled `.pyc` bytecode** baked into the Docker image at build time. The image was built from a slightly older commit than what is now on disk. When `apply_phase34_migrations` ran at startup it created the correct tables, but stale bytecode in the image still references columns from a pre-migration schema (`has_adapter` as a DB column rather than a computed field, `benchmark_name` instead of `benchmark`, `cron_schedule` instead of `trigger_config`, `ts` instead of `created_at`). These errors surface at whatever point those legacy code paths are invoked (the 2026-05-14 16:34 cluster likely from a UI page load that hit old route code).

**Problem 2 â€” Champion `run-d4dac705` not linked to `trading-reflector` track.** The `evolution_runs` row exists (status=completed), the `generations` row exists (promoted=true, is_champion=true), and the file-backed `registry.json` correctly names it champion. But `evolution_tracks.champion_run_id` still points to the earlier `run-ed119df1`. The runner's track auto-promotion block (`runner.py:412â€“418`) wrapped in a bare `try/except` that logs only a warning â€” if `update_track_champion` raised (likely because `evolution_tracks` was empty at the time of the run, before tracks were registered via `modelforge_register_tracks.py`), the failure was swallowed. The file-based `ModelRegistry().set_champion()` call at `runner.py:398` succeeded (registry.json is correct) but the DB write did not.

### Phase 1 â€” Fix the stale bytecode (fixes all 5 column errors)

The fix is a **container rebuild**, not a schema migration. The schema is already correct.


After restart, `init_db()` will re-run `_SCHEMA_STATEMENTS` (all `CREATE TABLE IF NOT EXISTS` â€” safe idempotent), then `apply_phase34_migrations()` will re-run its ALTER/CREATE block (also idempotent). The stale `.pyc` files will be replaced with bytecode compiled from the current source.

### Phase 2 â€” Backfill `run-d4dac705` into `evolution_tracks` and `track_generations`

After Phase 1 (schema confirmed correct), use the existing `/api/forge/sync_tracks` endpoint â€” it is exactly designed for this:

```bash

  -H "X-API-Key: $MODELFORGE_API_KEY" | python3 -m json.tool
```

This endpoint (`forge.py:112â€“193`) walks all promoted generations, picks the best by track's `target_benchmarks`, and calls `update_track_champion` for any improvement. Because `run-d4dac705` and `run-ed119df1` have identical scores (avg 0.125 on `trading-reflector`'s benchmarks), `sync_tracks` will **not** overwrite the `ed119df1` pointer (it only promotes strictly better champions: `if best_avg <= prev_avg: continue`).

If you want `run-d4dac705` as the canonical champion pointer (it completed later), the correct path is a direct API call:

